# Appendix R. Production Requirement Traceability

## How to Use This Matrix

This appendix starts from the book's production promises and points to the
evidence that makes each promise reviewable. Use it when you want to ask:

```text
The book says a reliable agent should have this control.
Where is it taught?
Where is it implemented?
Which test, query, or readiness gate proves it?
What reviewer question should still be asked?
```

This is not a marketing checklist. A requirement is covered only when it has:

```text
teaching chapter -> implementation artifact -> validation evidence -> reviewer question
```

If a row cannot name all four, the requirement is not yet production evidence.

## Requirement Review Protocol

Use this form during a chapter review, launch review, or external critique:

```text
requirement:
chapter evidence:
implementation evidence:
validation evidence:
operator evidence:
gap:
owner:
next artifact:
```

The important field is `gap`. A gap is not a failure if it is named. It becomes
dangerous when the book or system implies a requirement is satisfied without
evidence.

## No-Evidence No-Claim Rule

When improving the book, do not add a reliability claim unless at least one of
these is true:

```text
the chapter points to a checked Rust type
the chapter points to a checked SQL file
the chapter points to a runbook query
the chapter points to a test or readiness gate
the chapter clearly marks the topic as future scaling, not main-path evidence
```

This rule keeps the book from drifting into hype. The claim can be ambitious,
but the evidence must be inspectable.

## Summary

Traceability is how a technical book earns trust. The reader should not have to
wonder whether "typed tools," "Postgres-backed scheduling," "human approval,"
"observability," or "recovery" are slogans. Each requirement should point to a
chapter, a source artifact, a validation command, and a reviewer question. That
chain is what turns the book from a set of ideas into a production manual.
